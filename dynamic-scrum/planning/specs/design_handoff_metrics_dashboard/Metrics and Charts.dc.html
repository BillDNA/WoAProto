1bPer-turn trace — the one new capture that feeds every temporal metric
Trace row shape (woa.db, per battle)
{ v:"1.2", map:"frontier", seed:81103, fp:"red",
  winner:"blue", winType:"attrition", turns:29,
  trace:[            // one entry per play, in order
    { t:1,  s:"red",  a:"deploy" },
    { t:2,  s:"blue", a:"deploy" },
    { t:9,  s:"red",  a:"attack", k:1, ld:"red" },
    { t:10, s:"blue", a:"swap" },
    …                // a: deploy|attack|swap|march
  ],                 // k: kills, ld: leader after turn
  units:{ cav:{dep:[3,5], atk:6, kill:3, die:1},
          inf:{…}, art:{…} } }
Why this shape: deploy interleave, first contact, settle point, kill tempo, Drag, Swings, action shares and the unit buckets are all folds over the trace — the metric definitions move into report-model.js and stay one-implementation-per-fact. Scalars in balanceAdd become derived, not separately counted.

Cost: ~32 plays × ~30 B ≈ 1 KB/battle → a 700-battle run ≈ 0.7 MB. Fine for woa.db; markdown reports keep printing folded numbers only.

Capture point: extend the existing playLog (it already records per-play card events) rather than a second log — add action/kill/leader fields to what's there.
