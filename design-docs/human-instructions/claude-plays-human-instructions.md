# Claude Plays â€” human instructions

`dev/claude-plays.js` runs one battle of War of Attrition where an LLM (via the
`claude -p` subscription CLI, see `dev/llm-client.js`) plays one or both sides.
This doc is the operator's manual: how to run it, what it sees, and how to read
the report it leaves behind.

## What it's for

Heuristic AI (`easy`/`normal`/`hard` in `balance.js`) is fast and good for bulk
balance stats, but it never tells you *why* a card or map felt bad. Claude Plays
trades speed for judgment: it plays like a person reading the rules for the
first time, explains each pick in one sentence, and gives free-text "how did
that feel" notes at the end. Use it when a metric looks off and you want a
plausible human reason, not just a number.

## Running a battle

```
node dev/claude-plays.js [options]
```

| Option | Default | Meaning |
|---|---|---|
| `--map <filter>` | first built-in map | case-insensitive substring match against map names |
| `--red <spec>` | `haiku` | `easy`/`normal`/`hard` = heuristic AI; anything else (`haiku`/`sonnet`/`opus`/a full model id) = LLM |
| `--blue <spec>` | `normal` | same as `--red` |
| `--effort <level>` | CLI default | `low`/`medium`/`high`/`xhigh`/`max` reasoning effort for the LLM sides' `claude -p` calls (added round 2) |
| `--seed <n>` | `1234` | match seed â€” reuse a seed to replay the same deal |
| `--max-turns <n>` | `60` | safety cap; the run reports "unfinished" if it hits this |
| `--mock` | off | swaps the LLM transport for a canned deterministic fake â€” no `claude` process spawned, useful for testing the harness itself |
| `--out <file>` | `design-docs/game-logs/claude-plays-log.jsonl` | where the JSON-lines record is appended |

Examples:
```
node dev/claude-plays.js --map thornfield --blue sonnet --red hard --effort high
node dev/claude-plays.js --red haiku --blue haiku --seed 42
node dev/claude-plays.js --mock          # offline smoke test, no LLM calls
```

Every decision prints live to the console as the battle plays out, prefixed
`T<turn> <side> [<model>]:`. Fallback picks (see below) are flagged inline.

**Setting the map** â€” pass `--map <text>`; it matches against `E.MAPS[].name`
case-insensitively. Leave it off to use the first built-in map.

**Setting the effort level** â€” pass `--effort <low|medium|high|xhigh|max>`
(added round 2). It's threaded straight to the `--effort` flag of every
`claude -p` call the LLM sides make, so higher = more deliberation per move
(and per felt-note). Omit it to use the CLI's own default. It has no effect on
a heuristic side (`easy`/`normal`/`hard`).

## What the LLM actually sees

Each side is given a system prompt with the full rules (a distilled version of
`game/README.md`, baked into `claude-plays.js` as `RULES`) plus its assigned
side. Per decision it receives:

- **Honest state only** â€” its own units, reserves, spent-card pile, and hand
  (when picking a card); the opponent's units, reserves, spent pile, and
  hidden hand **count** only, never their hand contents or deck order.
- **A numbered list of legal moves**, generated by the engine itself
  (`playCard` modes / `enumerateChoices`) â€” the LLM can never invent an illegal
  move, because it can only pick an index into a list the rules engine built.
- For attack options, a **precomputed power preview** (attacker vs. defender
  power, and the outcome) so the model doesn't have to do hex-adjacency math
  in its head.

The model replies with `{choice: <index>, why: "<one sentence>"}` (JSON,
enforced via a schema appended to the prompt â€” see `CHOICE_SCHEMA` /
`buildPrompt` in `llm-client.js`).

## Fail-open behaviour

Any bad response â€” a spawn failure, timeout, malformed JSON, an out-of-range
index â€” never crashes the run. It falls back to a safe default and the
decision is logged with `fallback: true`:

- **Card-pick fallback**: use the heuristic engine's own `aiPlanTurn('normal')` choice.
- **Step-pick fallback**: take the first legal option; if that one is rejected
  (e.g. the must-play-step rule forbids skipping this step), fall to the next
  legal option until one applies â€” it never retries a bad skip and hangs.

A run's fallback rate (`fallbacks` in the JSONL record) is itself a signal â€”
a model that fails open often is one you can't trust for a real reading of
"how it felt."

## After the battle: felt-notes

Once the battle ends (HQ capture or attrition at deck-out), each *LLM* player
(heuristic sides are skipped) is shown the full campaign journal and asked for
under-150-word notes: what felt strong, what felt weak, what felt
luck-driven, and one suggested change. This is the qualitative payoff â€”
read it before the numbers.

## The logs: `design-docs/game-logs/`

Two artifacts land there per run (round 2 moved them out of `dev/`):

- **`claude-plays-log.jsonl`** â€” the machine record, one JSON object per line
  (pass `--out` to redirect). Fields:

  | Field | Meaning |
  |---|---|
  | `ts`, `map`, `seed`, `red`, `blue`, `effort` | run identity |
  | `winner`, `winType`, `turns` | outcome (`winType`: `hq` or `attrition`; `winner: null` + `winType: 'unfinished'` if it hit `--max-turns`) |
  | `fallbacks` | count of forced-default decisions (see above) |
  | `decisions` | full move-by-move list: `{turn, side, kind, choice, why, fallback}` |
  | `notes` | `{red?, blue?}` felt-notes text, only for LLM-controlled sides |
  | `usage` | summed `inputTokens`/`outputTokens` across the whole run |

- **`<timestamp>-<map>-<red>-v-<blue>.md`** â€” a readable per-run transcript:
  result, decision list, campaign journal, and felt-notes. Eyeball this one.

To turn a pile of logs into a one-page suggestions summary, run the
**`game-log-report`** skill ("read the game logs" / "game-log report") â€” it
reads this folder and returns scope, outcomes, distilled felt-notes, ranked
suggestions, and a watch-list. (That's the summarizer the round-2 feedback
asked for; it now exists.)

## Gotchas

- Needs the `claude` CLI installed and authenticated on this machine â€”
  `llm-client.js` resolves the real binary itself (handles the Windows
  `.cmd` shim case) and fails open if it's missing, so a broken install shows
  up as 100% fallbacks, not a crash.
- Every LLM turn is a *cold* `claude -p` process â€” no conversation memory
  between decisions within a battle. Each call restates the full state.
- `--mock` never spawns a real process; use it to verify the harness (turn
  loop, logging, JSONL shape) without burning API calls.
